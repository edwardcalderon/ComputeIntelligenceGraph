# Requirements Document: CIG Authentication & Provisioning

## Introduction

This document specifies the authentication and provisioning milestone for the CIG (Compute Intelligence Graph) platform. CIG operates in two distinct modes — managed cloud and self-hosted — and this milestone establishes a clean, mode-aware authentication architecture that avoids forcing Authentik as a universal dependency, avoids reusing human tokens as node identity, and avoids treating bootstrap tokens as permanent credentials.

The existing codebase has scaffolded CLI commands, a Supabase-backed auth package, an AuthentikDeployer in packages/infra, and credential storage in the CLI. This milestone wires those pieces together and fills the gaps: the Authentik AWS infrastructure blueprint, OIDC integration, the CLI device-code login flow, enrollment, self-hosted bootstrap, install profiles, and the corresponding API endpoints and dashboard pages.

## Glossary

- **CIG**: Compute Intelligence Graph — the self-hosted infrastructure intelligence platform
- **Managed_Mode**: Operational mode where CIG connects to the CIG cloud control plane; Authentik is the identity provider for human users
- **Self_Hosted_Mode**: Operational mode where CIG runs entirely on local or private infrastructure; no Authentik dependency
- **Authentik**: Open-source identity provider deployed to AWS for Managed_Mode human authentication
- **OIDC_Provider**: Authentik acting as an OpenID Connect identity provider for the Dashboard and API in Managed_Mode
- **Device_Authorization_Grant**: OAuth 2.0 Device Authorization Grant (RFC 8628) used by the CLI to authenticate without a browser redirect
- **Enrollment_Token**: Short-lived, single-use token issued by the CIG API after CLI login; used to register a Target_Node
- **Target_Node**: A physical or virtual machine where CIG services are installed and managed
- **Node_Identity**: Persistent Ed25519 key pair assigned to a Target_Node at enrollment time; distinct from human user tokens
- **Install_Manifest**: Configuration bundle returned by the API after enrollment; contains service definitions, secrets, and Node_Identity credentials
- **Bootstrap_Token**: One-time secret generated by the CLI for Self_Hosted_Mode; consumed on first admin account creation
- **Install_Profile**: Predefined service set — `core` (API, dashboard, graph, discovery, Cartography, auth/bootstrap) or `full` (core + chatbot + agents + vector DB)
- **Heartbeat**: Periodic status message sent by an enrolled Target_Node to the CIG API to report liveness and service health
- **API**: The CIG Fastify REST/GraphQL/WebSocket server (packages/api)
- **Dashboard**: The CIG Next.js web application (apps/dashboard)
- **CLI**: The CIG command-line tool (packages/cli, `cig` binary)
- **Auth_Package**: The shared authentication helpers package (packages/auth)
- **Infra_Package**: The AWS deployment wrapper package (packages/infra)
- **IAC_Package**: The Terraform modules package (packages/iac)
- **Control_Plane**: The CIG cloud API that manages enrolled Target_Nodes in Managed_Mode
- **Session**: A short-lived authenticated context for a human user in the Dashboard or API
- **Internal_JWT**: A short-lived JSON Web Token issued by the CIG API for internal service-to-service calls

## Requirements

### Requirement 1: Authentik AWS Infrastructure Blueprint

**User Story:** As a platform engineer, I want a complete Authentik deployment blueprint for AWS, so that I can provision a production-grade identity provider for Managed_Mode without manual configuration.

#### Acceptance Criteria

1. THE Infra_Package SHALL provide an `AuthentikInfraBlueprint` that provisions a VPC with public and private subnets across two availability zones
2. THE Infra_Package SHALL provision a managed PostgreSQL instance (AWS RDS) as the Authentik database backend
3. THE Infra_Package SHALL deploy Authentik as an ECS Fargate service behind an Application Load Balancer
4. THE Infra_Package SHALL provision an ACM certificate and configure HTTPS termination on the ALB
5. THE Infra_Package SHALL configure Route 53 DNS records pointing the Authentik domain to the ALB
6. THE Infra_Package SHALL store the Authentik admin password and database credentials in AWS Secrets Manager
7. WHEN the blueprint deployment completes, THE Infra_Package SHALL output the Authentik issuer URL, OIDC client ID, and OIDC client secret as structured outputs
8. WHEN the blueprint deployment fails at any phase, THE Infra_Package SHALL roll back provisioned resources and return a descriptive error
9. WHERE the `vpcId` option is provided, THE Infra_Package SHALL deploy into the existing VPC instead of creating a new one
10. THE IAC_Package SHALL contain a Terraform module `modules/authentik-aws` that the Infra_Package references for all Authentik infrastructure resources

### Requirement 2: OIDC Integration for Dashboard and API

**User Story:** As a platform engineer, I want the Dashboard and API to support OIDC login via Authentik in Managed_Mode while remaining fully functional in Self_Hosted_Mode, so that human users can authenticate without breaking local deployments.

#### Acceptance Criteria

1. WHEN `CIG_AUTH_MODE=managed` is set, THE Auth_Package SHALL expose an OIDC adapter that validates ID tokens issued by the configured Authentik issuer
2. WHEN `CIG_AUTH_MODE=self-hosted` is set, THE Auth_Package SHALL expose a local JWT adapter that validates tokens issued by the CIG API itself
3. THE Auth_Package SHALL export a `createAuthAdapter(mode, config)` factory that returns the correct adapter without requiring callers to import Supabase or Authentik directly
4. WHEN the Dashboard loads in Managed_Mode, THE Dashboard SHALL redirect unauthenticated users to the Authentik OIDC authorization endpoint
5. WHEN Authentik returns an authorization code, THE Dashboard SHALL exchange it for tokens via the API's `/api/v1/auth/oidc/callback` endpoint
6. THE API SHALL validate the OIDC ID token signature against the Authentik JWKS endpoint on every authenticated request in Managed_Mode
7. THE API SHALL extract `sub`, `email`, `groups`, and `tenant` claims from the validated ID token and attach them to the request context
8. WHEN the Dashboard loads in Self_Hosted_Mode, THE Dashboard SHALL redirect unauthenticated users to the local login page instead of Authentik
9. THE Auth_Package SHALL not import or instantiate any Supabase client when `CIG_AUTH_MODE` is not `supabase`
10. WHEN an OIDC token is expired, THE API SHALL return HTTP 401 with error code `token_expired`
11. WHEN an OIDC token signature is invalid, THE API SHALL return HTTP 401 with error code `token_invalid`

### Requirement 3: CLI Device Authorization Login Flow

**User Story:** As a CIG user, I want to run `cig login` and authenticate via a device code displayed in my terminal, so that I can securely connect the CLI to my CIG account without pasting tokens.

#### Acceptance Criteria

1. WHEN a user executes `cig login`, THE CLI SHALL POST to `/api/v1/auth/device/authorize` and receive a `device_code`, `user_code`, `verification_uri`, and `expires_in`
2. WHEN the device authorization response is received, THE CLI SHALL display the `user_code` and `verification_uri` prominently in the terminal
3. WHEN the device authorization response is received, THE CLI SHALL begin polling `/api/v1/auth/device/poll` with the `device_code` at 5-second intervals
4. THE CLI SHALL apply exponential backoff on poll intervals up to a maximum of 30 seconds when the API returns `slow_down`
5. WHEN the poll response returns `approved` status, THE CLI SHALL store the returned `access_token` and `refresh_token` using the CredentialManager with AES-256-GCM encryption
6. WHEN the poll response returns `denied` status, THE CLI SHALL display an error message and exit with code 1
7. WHEN the `device_code` expires before approval (15 minutes), THE CLI SHALL display an expiry message and exit with code 1
8. WHEN a user executes `cig logout`, THE CLI SHALL delete all stored tokens and Node_Identity credentials via CredentialManager and POST to `/api/v1/auth/logout`
9. WHEN the stored `access_token` is within 5 minutes of expiry, THE CLI SHALL automatically refresh it using the `refresh_token` before making API calls
10. WHEN the `refresh_token` is expired, THE CLI SHALL prompt the user to run `cig login` again

### Requirement 4: Enrollment Token and Install Manifest

**User Story:** As a CIG user, I want the CLI to obtain an enrollment token and install manifest from the API after login, so that my Target_Node receives a unique identity and correct service configuration.

#### Acceptance Criteria

1. WHEN `cig install` runs in Managed_Mode with a valid session, THE CLI SHALL POST to `/api/v1/targets/enrollment-token` to obtain an Enrollment_Token
2. THE Enrollment_Token SHALL be a UUID, valid for 10 minutes, and single-use
3. WHEN the CLI holds a valid Enrollment_Token, THE CLI SHALL POST to `/api/v1/targets/enroll` with the token and Target_Node metadata (hostname, OS, architecture, IP address)
4. WHEN the API validates the Enrollment_Token, THE API SHALL generate a Node_Identity consisting of a UUID `target_id` and an Ed25519 key pair
5. WHEN the Node_Identity is generated, THE API SHALL return the `target_id`, `private_key`, and `public_key` to the CLI
6. THE CLI SHALL store the Node_Identity in `~/.cig/auth.json` using the CredentialManager with permissions 0600
7. WHEN enrollment succeeds, THE CLI SHALL GET `/api/v1/targets/install-manifest?target_id=<id>&profile=<profile>` to retrieve the Install_Manifest
8. THE Install_Manifest SHALL include the selected Install_Profile service list, generated secrets, environment variable overrides, and the Node_Identity credentials
9. WHEN the Enrollment_Token has expired, THE CLI SHALL request a new token and retry enrollment once
10. THE Node_Identity private key SHALL never be transmitted back to the API after initial issuance

### Requirement 5: Self-Hosted Bootstrap Flow

**User Story:** As a CIG user, I want to install CIG in Self_Hosted_Mode without a CIG account, so that I can run CIG entirely on my own infrastructure with no cloud dependency.

#### Acceptance Criteria

1. THE CLI SHALL accept a `--mode self-hosted` flag on the `cig install` command
2. WHEN `--mode self-hosted` is specified, THE CLI SHALL skip all authentication, device authorization, and enrollment steps
3. WHEN `--mode self-hosted` is specified, THE CLI SHALL generate a cryptographically random 32-character Bootstrap_Token
4. THE CLI SHALL store the Bootstrap_Token in `~/.cig/bootstrap.json` with permissions 0600
5. WHEN installation completes in Self_Hosted_Mode, THE CLI SHALL display the local Dashboard URL and the Bootstrap_Token
6. WHEN a user accesses the Dashboard for the first time in Self_Hosted_Mode, THE Dashboard SHALL detect that no admin accounts exist and display the bootstrap setup page
7. WHEN the user submits a valid Bootstrap_Token on the bootstrap setup page, THE Dashboard SHALL display the admin account creation form
8. THE admin account creation form SHALL require a username, email address, and password of at least 12 characters
9. WHEN the admin account form is submitted, THE API SHALL validate the Bootstrap_Token via `POST /api/v1/bootstrap/complete`
10. WHEN the Bootstrap_Token is valid, THE API SHALL create the admin account, invalidate the Bootstrap_Token, and return a session token
11. WHEN the Bootstrap_Token has been consumed, THE API SHALL reject any further bootstrap requests with HTTP 409
12. WHEN the Bootstrap_Token is expired (30 minutes after generation), THE CLI SHALL provide a `cig bootstrap-reset` command to generate and display a new token
13. THE API SHALL expose `GET /api/v1/bootstrap/status` that returns `{ requires_bootstrap: true }` when no admin accounts exist

### Requirement 6: Install Profiles

**User Story:** As a CIG user, I want to choose between `core` and `full` install profiles, so that I can deploy only the services I need without unnecessary resource consumption.

#### Acceptance Criteria

1. THE CLI SHALL accept a `--profile` flag on `cig install` with values `core` or `full`; the default SHALL be `core`
2. WHEN the `core` profile is selected, THE CLI SHALL include exactly these services in the generated compose stack: API, Dashboard, Neo4j, Discovery, Cartography, and the auth/bootstrap service
3. WHEN the `full` profile is selected, THE CLI SHALL include all `core` services plus Chatbot, Agents, and Chroma vector database
4. WHEN no `--profile` flag is provided, THE CLI SHALL interactively prompt the user to select a profile and display the service list for each option
5. WHEN the user confirms a profile, THE CLI SHALL display the complete list of services that will be installed before proceeding
6. THE Install_Manifest returned by the API SHALL include the resolved service list for the requested profile

### Requirement 7: CLI Install Flow

**User Story:** As a CIG user, I want `cig install` to handle all prerequisite checks, secret generation, service startup, and health verification, so that I can install CIG reliably without manual steps.

#### Acceptance Criteria

1. WHEN `cig install` executes, THE CLI SHALL verify that Docker Engine is installed and the Docker daemon is running
2. WHEN `cig install` executes, THE CLI SHALL verify that Docker Compose v2.0 or later is available
3. WHEN `cig install` executes, THE CLI SHALL verify that at least 4 GB of free memory is available on the host
4. WHEN `cig install` executes, THE CLI SHALL verify that at least 10 GB of free disk space is available
5. WHEN `cig install` executes, THE CLI SHALL verify that ports 3000, 7474, 7687, 8000, and 8080 are not in use
6. WHEN any prerequisite check fails, THE CLI SHALL display a clear error message with remediation steps and exit with code 1
7. WHEN all prerequisite checks pass, THE CLI SHALL generate a `docker-compose.yml` and `.env` file in the installation directory based on the Install_Profile and Install_Manifest
8. THE generated `.env` file SHALL contain cryptographically random passwords for all services and SHALL be written with permissions 0600
9. WHEN the compose stack is ready, THE CLI SHALL execute `docker compose up -d` and display per-container startup progress
10. WHEN a container fails to start, THE CLI SHALL display the container logs and offer to rollback
11. WHEN all containers are running, THE CLI SHALL poll each service health endpoint until all pass or a 5-minute timeout is reached
12. WHEN health checks pass, THE CLI SHALL write installation state to `~/.cig/state.json` with permissions 0600 and display the Dashboard URL
13. WHEN installation fails after services have started, THE CLI SHALL offer automatic rollback that stops and removes all CIG containers and generated config files
14. THE CLI SHALL provide a `cig doctor` command that runs all prerequisite checks and displays a readiness summary without installing

### Requirement 8: Target Node Heartbeat

**User Story:** As a CIG operator, I want enrolled Target_Nodes to send periodic heartbeats to the Control_Plane, so that I can monitor node liveness and service health from the Dashboard.

#### Acceptance Criteria

1. WHEN a Target_Node is installed in Managed_Mode, THE installed API service SHALL POST to `/api/v1/targets/:id/heartbeat` every 60 seconds using the Node_Identity private key to sign the request
2. THE heartbeat request body SHALL include `service_status` (map of service name to health status) and `system_metrics` (CPU usage percentage, memory usage percentage, disk usage percentage)
3. THE heartbeat request SHALL include an `X-CIG-Signature` header containing an Ed25519 signature of the request body signed with the Node_Identity private key
4. WHEN the API receives a heartbeat, THE API SHALL verify the Ed25519 signature against the stored public key for the `target_id`
5. WHEN the signature is valid, THE API SHALL update the Target_Node's `last_seen` timestamp and store the latest metrics
6. WHEN the signature is invalid, THE API SHALL return HTTP 401 and increment a signature failure counter for the Target_Node
7. WHEN a Target_Node has not sent a heartbeat for 5 minutes, THE API SHALL set the node status to `degraded`
8. WHEN a Target_Node has not sent a heartbeat for 15 minutes, THE API SHALL set the node status to `offline`
9. WHEN a Target_Node reconnects after being `offline`, THE API SHALL set the node status to `online`
10. WHEN the heartbeat connection fails, THE installed API service SHALL retry with exponential backoff starting at 5 seconds, doubling up to a maximum of 60 seconds

### Requirement 9: Dashboard — Device Approval Page

**User Story:** As a CIG user, I want to approve CLI device authorization requests from the Dashboard, so that I can securely authorize a new CLI session without leaving the browser.

#### Acceptance Criteria

1. WHEN a CLI initiates device authorization, THE API SHALL create a pending device authorization record associated with the `user_code`
2. WHEN an authenticated user opens the Dashboard, THE Dashboard SHALL display a notification badge if there are pending device authorization requests for their account
3. WHEN the user navigates to the device approval page, THE Dashboard SHALL display each pending request showing the `user_code`, requesting IP address, and request timestamp
4. WHEN the user clicks "Approve" on a pending request, THE Dashboard SHALL POST to `/api/v1/auth/device/approve` with the `user_code`
5. WHEN the API receives an approval, THE API SHALL mark the device as approved and make tokens available on the next poll
6. WHEN the user clicks "Deny" on a pending request, THE Dashboard SHALL POST to `/api/v1/auth/device/deny` with the `user_code`
7. WHEN a device authorization record expires (15 minutes), THE Dashboard SHALL automatically remove it from the pending list without a page reload
8. WHEN the user approves or denies a request, THE Dashboard SHALL remove it from the pending list immediately

### Requirement 10: Dashboard — Managed Targets Page

**User Story:** As a CIG user, I want a Targets page in the Dashboard that lists all enrolled Target_Nodes, so that I can monitor their status and manage their access.

#### Acceptance Criteria

1. THE Dashboard SHALL provide a `/targets` page that lists all Target_Nodes enrolled under the authenticated user's account
2. WHEN the Targets page loads, THE Dashboard SHALL display each node's hostname, operating system, install profile, status, and last heartbeat timestamp
3. WHEN a Target_Node status is `online`, THE Dashboard SHALL display a green status indicator
4. WHEN a Target_Node status is `degraded`, THE Dashboard SHALL display a yellow status indicator
5. WHEN a Target_Node status is `offline`, THE Dashboard SHALL display a red status indicator
6. WHEN the user clicks a Target_Node row, THE Dashboard SHALL display a detail panel with IP address, architecture, installed services, CIG version, and uptime
7. THE Dashboard SHALL provide a "Revoke Access" button per node that POSTs to `/api/v1/targets/:id/revoke`
8. WHEN the API receives a revoke request, THE API SHALL invalidate the Node_Identity and set the node status to `revoked`
9. WHEN a node is revoked, THE Dashboard SHALL update the node status to `revoked` without a full page reload

### Requirement 11: Dashboard — Self-Hosted Bootstrap Page

**User Story:** As a CIG user setting up a self-hosted instance, I want a first-run bootstrap page in the Dashboard, so that I can create the initial admin account securely using the Bootstrap_Token.

#### Acceptance Criteria

1. WHEN the Dashboard loads and `GET /api/v1/bootstrap/status` returns `requires_bootstrap: true`, THE Dashboard SHALL redirect all unauthenticated routes to `/bootstrap`
2. WHEN the `/bootstrap` page loads, THE Dashboard SHALL display a Bootstrap_Token input field and instructions to retrieve the token from the CLI output
3. WHEN the user submits a Bootstrap_Token, THE Dashboard SHALL POST to `/api/v1/bootstrap/validate` to verify the token before showing the account creation form
4. WHEN the token is valid, THE Dashboard SHALL display the admin account creation form with fields for username, email, and password
5. WHEN the user submits the admin account form, THE Dashboard SHALL POST to `/api/v1/bootstrap/complete` with the token and account details
6. WHEN bootstrap completes successfully, THE Dashboard SHALL redirect the user to the login page
7. WHEN the Bootstrap_Token is invalid or expired, THE Dashboard SHALL display an error message with instructions to run `cig bootstrap-reset`
8. WHEN bootstrap is complete (no longer required), THE Dashboard SHALL not render the `/bootstrap` page and SHALL redirect to the login page

### Requirement 12: API — Device Authorization Endpoints

**User Story:** As a CIG API developer, I want device authorization endpoints that implement RFC 8628, so that the CLI can authenticate users without a browser redirect.

#### Acceptance Criteria

1. THE API SHALL provide `POST /api/v1/auth/device/authorize` that generates a `device_code` (32 hex characters), a `user_code` (8 alphanumeric characters), and stores them with a 15-minute expiry
2. THE authorize endpoint SHALL return `{ device_code, user_code, verification_uri, expires_in: 900 }`
3. THE API SHALL provide `POST /api/v1/auth/device/poll` that accepts `{ device_code }` and returns `{ status: "pending" | "approved" | "denied" | "expired" }`
4. WHEN status is `approved`, THE poll endpoint SHALL also return `{ access_token, refresh_token, token_type: "Bearer" }`
5. THE API SHALL provide `POST /api/v1/auth/device/approve` that accepts `{ user_code }` from an authenticated Dashboard session and marks the device as approved
6. THE API SHALL provide `POST /api/v1/auth/device/deny` that accepts `{ user_code }` from an authenticated Dashboard session and marks the device as denied
7. THE poll endpoint SHALL be rate-limited to 1 request per 5 seconds per `device_code`; exceeding this SHALL return `{ status: "slow_down" }`
8. WHEN a `device_code` is expired or unknown, THE poll endpoint SHALL return `{ status: "expired" }`
9. THE API SHALL provide `POST /api/v1/auth/logout` that invalidates the session associated with the provided Bearer token

### Requirement 13: API — Target Enrollment Endpoints

**User Story:** As a CIG API developer, I want target enrollment endpoints, so that the CLI can register Target_Nodes and receive Node_Identity credentials.

#### Acceptance Criteria

1. THE API SHALL provide `POST /api/v1/targets/enrollment-token` that requires a valid Bearer token and returns `{ enrollment_token, expires_at }`
2. THE Enrollment_Token SHALL be a UUID, stored with a 10-minute expiry, and marked single-use
3. THE API SHALL provide `POST /api/v1/targets/enroll` that accepts `{ enrollment_token, hostname, os, architecture, ip_address }`
4. WHEN the Enrollment_Token is valid, THE API SHALL generate a `target_id` (UUID) and an Ed25519 key pair, store the public key, and return `{ target_id, private_key, public_key }`
5. WHEN the Enrollment_Token is expired or already used, THE API SHALL return HTTP 410 with error code `token_consumed`
6. THE API SHALL provide `GET /api/v1/targets` that returns the list of Target_Nodes for the authenticated user
7. THE API SHALL provide `DELETE /api/v1/targets/:id` (alias: revoke) that invalidates the Node_Identity and sets status to `revoked`
8. THE API SHALL provide `GET /api/v1/targets/install-manifest` that accepts `target_id` and `profile` query parameters and returns the Install_Manifest JSON
9. THE Install_Manifest SHALL include `profile`, `services`, `env_overrides`, `node_identity`, and `generated_secrets` fields

### Requirement 14: API — Heartbeat Endpoint

**User Story:** As a CIG API developer, I want a heartbeat endpoint that validates Node_Identity signatures, so that only legitimately enrolled nodes can report status.

#### Acceptance Criteria

1. THE API SHALL provide `POST /api/v1/targets/:id/heartbeat` that requires an `X-CIG-Signature` header
2. WHEN the heartbeat endpoint receives a request, THE API SHALL verify the Ed25519 signature in `X-CIG-Signature` against the stored public key for `:id`
3. WHEN the signature is valid, THE API SHALL update `last_seen`, store `service_status` and `system_metrics`, and return HTTP 200
4. WHEN the signature is invalid, THE API SHALL return HTTP 401 with error code `invalid_signature`
5. THE heartbeat endpoint SHALL be rate-limited to 1 request per 30 seconds per `target_id`
6. THE API SHALL emit a WebSocket event `target:heartbeat` to connected Dashboard clients when a heartbeat is received
7. THE API SHALL run a background job every 60 seconds that sets nodes with `last_seen` older than 5 minutes to `degraded` and older than 15 minutes to `offline`

### Requirement 15: API — Bootstrap Endpoints

**User Story:** As a CIG API developer, I want bootstrap endpoints for self-hosted installations, so that the first admin account can be created securely using a one-time token.

#### Acceptance Criteria

1. THE API SHALL provide `GET /api/v1/bootstrap/status` that returns `{ requires_bootstrap: boolean }` based on whether any admin accounts exist
2. THE API SHALL provide `POST /api/v1/bootstrap/validate` that accepts `{ bootstrap_token }` and returns `{ valid: true }` or HTTP 401
3. THE API SHALL provide `POST /api/v1/bootstrap/complete` that accepts `{ bootstrap_token, username, email, password }`
4. WHEN the Bootstrap_Token is valid and no admin accounts exist, THE API SHALL create the admin account with full permissions and return `{ access_token, refresh_token }`
5. WHEN the Bootstrap_Token has already been consumed, THE API SHALL return HTTP 409 with error code `bootstrap_already_complete`
6. WHEN the Bootstrap_Token is expired, THE API SHALL return HTTP 401 with error code `bootstrap_token_expired`
7. WHEN the password is shorter than 12 characters, THE API SHALL return HTTP 422 with error code `password_too_short`
8. WHEN bootstrap completes, THE API SHALL write an audit event `bootstrap_complete` with the admin account email and timestamp
9. THE bootstrap endpoints SHALL only accept requests from localhost (`127.0.0.1` or `::1`) in Self_Hosted_Mode; requests from other origins SHALL return HTTP 403

### Requirement 16: API — OIDC Callback Endpoint

**User Story:** As a CIG API developer, I want an OIDC callback endpoint, so that the Dashboard can complete the Authentik authorization code exchange server-side.

#### Acceptance Criteria

1. THE API SHALL provide `GET /api/v1/auth/oidc/callback` that accepts `code` and `state` query parameters
2. WHEN the callback endpoint receives a valid `code`, THE API SHALL exchange it for tokens at the Authentik token endpoint
3. WHEN the token exchange succeeds, THE API SHALL validate the ID token signature against the Authentik JWKS endpoint
4. WHEN the ID token is valid, THE API SHALL create or update the user record from the `sub`, `email`, and `groups` claims
5. WHEN the user record is ready, THE API SHALL issue a CIG session token and redirect the Dashboard to the post-login route
6. WHEN the `state` parameter does not match the stored CSRF state, THE API SHALL return HTTP 400 with error code `invalid_state`
7. WHEN the Authentik token exchange fails, THE API SHALL return HTTP 502 with error code `oidc_upstream_error`

### Requirement 17: Internal Service Authentication

**User Story:** As a platform engineer, I want a minimal internal JWT mechanism for service-to-service calls, so that internal CIG services can authenticate with each other without using human user tokens or Authentik.

#### Acceptance Criteria

1. THE API SHALL issue Internal_JWTs signed with a shared HMAC-SHA256 secret loaded from the `CIG_INTERNAL_JWT_SECRET` environment variable
2. THE Internal_JWT SHALL have a maximum lifetime of 5 minutes and SHALL include `iss: "cig-internal"`, `sub: <service_name>`, and `exp` claims
3. WHEN an internal endpoint receives a request, THE API SHALL validate the Internal_JWT and reject tokens with `iss` other than `cig-internal`
4. THE Auth_Package SHALL export a `createInternalToken(serviceName, secret)` function and a `verifyInternalToken(token, secret)` function
5. THE internal JWT mechanism SHALL NOT use Authentik and SHALL NOT require network calls to any external identity provider

### Requirement 18: Audit Logging

**User Story:** As a security-conscious platform engineer, I want all authentication and provisioning events to be written to an audit log, so that I can investigate incidents and demonstrate compliance.

#### Acceptance Criteria

1. THE API SHALL write an audit event for each of the following actions: device authorization initiated, device approved, device denied, enrollment token issued, target enrolled, target revoked, heartbeat signature failure, bootstrap completed, admin login, and logout
2. EACH audit event SHALL include `event_type`, `actor` (user ID or target ID), `ip_address`, `timestamp` (ISO 8601), and `outcome` (`success` or `failure`)
3. THE API SHALL store audit events in a dedicated `audit_events` collection or table
4. THE API SHALL provide `GET /api/v1/audit` that returns paginated audit events for authenticated admin users
5. WHEN an audit write fails, THE API SHALL log the failure to the application logger but SHALL NOT block the primary operation
